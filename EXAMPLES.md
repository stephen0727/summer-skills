## 🎨 配图助手 (Image Assistant)

### 场景：为一篇技术文章配图

```
用户: 我写了一篇介绍 AI 工具选型的文章，需要配几张图
助手: [自动触发配图助手流程]
助手: 好的！让我先了解一下需求：
     1. 这篇文章主要用在哪里？（PPT、公众号、博客等）
     2. 谁来看？（技术人员、产品经理、普通用户？）
     3. 你希望图上是更简洁少字，还是信息密度高一些？

用户: 用在公众号文章里，面向产品经理和技术人员，希望简洁易读
助手: 明白了！让我看看文章内容...

     【第2步：配图规划】
     根据你的内容，我建议做 3 张图：
     1. 封面图：AI 工具选型的整体框架
     2. 对比图：开源 vs 商业方案的对比
     3. 决策流程图：如何选择合适的工具

     这样可以吗？

用户: 可以！
助手: 【第3步：文案定稿】
     第1张-封面图的文案：
     - 大标题：AI 工具选型指南
     - 副标题：场景→需求→选型
     - 3个关键词：性能、成本、生态

     确认这些文案吗？

[... 继续逐张确认文案 ...]

助手: 【第4步：提示词输出】
     第1张图的生图提示词：

\`\`\`
16:9横版信息图。奶油纸底+彩铅水彩手绘风格。
中心大标题【AI工具选型指南】，下方流程：场景分析→需求定义→工具选型。
底部3个图标+关键词：性能⚡、成本💰、生态🌱。
少字高可读，干净趣味。
\`\`\`

     可以直接复制提示词去生成图片了！需要调整吗？
```

**预期输出**:
- 图片清单（几张图/每张目的）
- 每张图的文案定稿（Copy Spec）
- 可复制的生图提示词
- （可选）批量生成的图片

## 🔄 需求变更工作流 (Requirement Change Workflow)

### 场景：修改 Chrome 扩展的侧边栏交互

```
用户: 我想改一下侧边栏的交互，现在点击按钮后直接跳转，我想加个确认弹窗
助手: [自动触发需求变更流程]
助手: 好的，让我们先明确一下这个变更的范围。

     【第1步：澄清变更】
     - 目标：在侧边栏按钮点击后，增加确认弹窗
     - 不改变的：跳转的目标页面、现有数据流
     - 验收标准：
       1. 点击按钮后显示确认弹窗
       2. 点击"确认"后执行原有跳转
       3. 点击"取消"后关闭弹窗

     这样理解对吗？

[... 继续执行基线确认、影响评估、设计提案等步骤 ...]
```

**预期输出**:
- 需求变更简报（change brief）
- 影响评估报告
- 实施后的代码变更
- 回归测试结果
- 决策日志

## 📦 需求池管理 (Backlog Manager)

### 场景 1：记录一个新想法

```
用户: 我想做个功能，让用户可以导出周报数据
助手: [自动触发需求池管理流程]
助手: 好的！让我先了解一下这个想法：
      1. 现在遇到什么问题了？什么场景下需要导出周报数据？
      2. 多久会碰到这个需求一次？
      3. 不做这个功能的话，你现在怎么解决？

用户: 现在周报数据只能在页面上看，领导想让我汇总月度数据，我只能手动复制粘贴
助手: 明白了！痛点是手动复制粘贴效率低，容易出错。
      我理解的是：需要一个导出功能，把周报数据导出为 Excel/CSV 方便汇总分析。
      对吗？

用户: 对的
助手: 好的，我将这个需求归档到"可以做了"区域。需要时随时可以启动开发。
```

**预期输出**:
- `docs/需求池.md` 新增条目
- 总览表更新
- 详情区记录痛点、方案、备注

### 场景 2：整理需求池

```
用户: 整理一下需求池
助手: 好的！让我看看当前的需求池状态...

     【逐条检查】

     #1 - 导出周报数据 (可以做了)
     建议：保持不变，痛点清晰，随时可以启动

     #2 - 暗黑模式 (先放着)
     建议：升档到"需要想想"。已收到 3 个用户反馈想要这个功能

     #3 - 接入 AI 自动写周报 (需要想想)
     建议：保持不变，还需要调研 API 成本和可行性

     #5 - 邮件通知功能 (可以做了)
     建议：降级到"先放着"。半年内没有用户提过这个需求

用户: #2 同意升档，#5 同意降级
助手: [更新需求池文件]
```

### 场景 3：挑选下一个版本做什么

```
用户: 下一个版本做什么
助手: 好的！让我看看"可以做了"的需求池...

     当前有 2 个候选：

     #1 - 导出周报数据
     #4 - 支持多团队切换

     让我用三个标准分析一下：

     | 标准 | 导出周报 | 多团队切换 |
     |------|---------|-----------|
     | 频率 | 每月 1-2 次 | 每天多次 |
     | 可绕过性 | 手动复制，很繁琐 | 必须登出重登，很麻烦 |
     | ROI | 中 | 高 |

     建议：优先做多团队切换，痛点频率高且无法绕过。
     但最终由你决定。

用户: 同意，先做 #4
助手: 好的！已将 #4 标记为"进行中"。需要我启动设计探索流程吗？
```

## 🔍 代码解释助手 (Explain Code)

### 场景：理解 Node.js 中的事件循环机制

```
用户: 帮我解释一下这段代码为什么输出顺序不是我预期的

      setTimeout(() => console.log('A'), 0)
      Promise.resolve().then(() => console.log('B'))
      console.log('C')

助手: [自动触发代码解释流程]
助手: 好！先打个比方——

      【类比】
      把 JavaScript 的事件循环想象成一家餐厅：
      - 你是主厨（主线程），一次只能做一道菜
      - 服务员（Promise）负责紧急加急单，放在你手边的「微任务队列」
      - 外卖小哥（setTimeout）负责普通取餐，放在门口等候区的「宏任务队列」

      规则：你每做完一道菜，先清空手边所有加急单，再去门口取一单外卖。

      【ASCII 图解】

      主线程执行
      ────────────────────────────────────
      1. console.log('C')        → 直接执行，输出 C
      2. setTimeout 注册         → 外卖小哥拿走，放门口等
      3. Promise.then 注册       → 服务员拿走，放手边桌
      ────────────────────────────────────
      主线程清空 → 处理微任务队列
      4. Promise.then 执行       → 输出 B
      ────────────────────────────────────
      微任务清空 → 处理宏任务队列
      5. setTimeout 执行         → 输出 A

      【输出顺序】：C → B → A

      【常见陷阱】
      ⚠️ setTimeout(fn, 0) 不是"立即执行"，而是"下一轮宏任务"
      ⚠️ Promise 是微任务，永远比 setTimeout 先执行
```

**预期输出**:
- 生活类比（降低理解门槛）
- ASCII 流程图（可视化执行顺序）
- 逐步执行说明
- 常见误解和陷阱提示
